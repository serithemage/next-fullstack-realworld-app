# ADR-007: 테스트 전략 (Jest + Playwright, 하이브리드 DB)

## Status

승인 (Accepted)

## Context

프로젝트에 테스트가 전무한 상태(0% 커버리지)에서, 바이브 코딩 전환의 안전망으로 테스트 체계를 구축해야 한다. 단위, 통합, E2E 세 레벨의 테스트가 필요하며, DB 연동 테스트의 전략이 핵심 결정 사항이다.

Prisma는 SQLite와 PostgreSQL 간 provider 호환이 되지 않아 동일 스키마를 두 DB에서 사용할 수 없다.

고려한 대안:
- **Vitest**: 빠르지만 Next.js 통합(next/jest)이 Jest 대비 미성숙
- **Cypress (E2E)**: 직관적 UI, 시간여행 디버깅이지만 Chromium 전용, 상대적으로 느림
- **SQLite in-memory**: 빠르지만 Prisma provider 호환 불가
- **모든 테스트에 Docker PostgreSQL**: 안정적이지만 단위 테스트가 느림

## Decision

**하이브리드 DB 테스트 전략을 채택한다.**

| 테스트 유형 | 프레임워크 | DB 전략 |
|------------|-----------|---------|
| 단위 테스트 | Jest + React Testing Library | Prisma Client mocking (jest-mock-extended) |
| 통합 테스트 | Jest | Docker PostgreSQL |
| E2E 테스트 | Playwright | Docker PostgreSQL + Seed 데이터 |
| CI | GitHub Actions | services 블록 PostgreSQL |

커버리지 목표: 60% 이상 (교육용 프로젝트에 현실적인 목표).

## Consequences

**긍정적:**
- 단위 테스트는 DB 없이 빠르게 실행 (mocking)
- 통합/E2E는 실제 PostgreSQL로 쿼리 정확성 검증
- Playwright는 Next.js 공식 권장, 다중 브라우저 지원
- E2E에서 기존 seed.ts의 테스트 사용자를 활용하여 안정적 테스트

**부정적:**
- 두 가지 DB 전략을 유지해야 하는 복잡성
- 통합/E2E 테스트는 Docker 의존으로 실행 환경 제약
- mocking 기반 단위 테스트는 실제 DB 동작과 차이가 있을 수 있음
