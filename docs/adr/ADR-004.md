# ADR-004: next-intl 국제화 전략

## Status

승인 (Accepted) — 2023-11-24

## Context

프로젝트에 다국어 지원(English, 中文简体)이 필요하다. Next.js App Router 환경에서 서버/클라이언트 컴포넌트 모두에서 번역을 사용해야 하며, URL에 locale이 반영되어야 한다.

고려한 대안:
- **next-i18next**: Pages Router에 최적화. App Router 지원 제한적
- **react-intl (FormatJS)**: 범용적이지만 Next.js 특화 기능 부재
- **직접 구현**: 번역 파일 로딩은 간단하나 라우팅 통합이 복잡

## Decision

**next-intl 3.x를 사용하며, `localePrefix: 'as-needed'` 설정을 적용한다.**

- App Router 네이티브 지원 (서버 컴포넌트에서 `getTranslations`, 클라이언트에서 `useTranslations`)
- `[locale]` 동적 세그먼트로 URL 기반 locale 라우팅
- `localePrefix: 'as-needed'`로 기본 locale(en)은 접두사 없이, 비기본 locale(zh)만 접두사 추가
- `src/navigation.ts`에서 `Link`, `redirect`, `usePathname`, `useRouter`를 re-export하여 locale 인식 네비게이션 제공

## Consequences

**긍정적:**
- App Router의 서버/클라이언트 컴포넌트 경계를 자연스럽게 처리
- Next.js 미들웨어 통합으로 locale 감지/리다이렉트 자동화
- `messages/{locale}.json`으로 번역 파일 관리 간편
- `NextIntlClientProvider`로 클라이언트 컴포넌트에 번역 전달

**부정적:**
- 미들웨어 체인에서 `withIntl`이 가장 마지막에 실행되어야 하는 순서 의존성
- locale이 URL 경로의 첫 세그먼트가 되어 다른 미들웨어(withFeed, withAuth)에서 locale 패턴을 인식해야 함
- 새 locale 추가 시 번역 파일 전체를 작성해야 함
