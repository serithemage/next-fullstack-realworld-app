# ADR-001: Next.js 14 App Router 선택

## Status

승인 (Accepted) — 2023-11-24

## Context

RealWorld 스펙(Medium.com 클론)을 구현하는 풀스택 웹 애플리케이션을 개발해야 한다. 서버 사이드 렌더링, API 라우트, 파일 기반 라우팅이 필요하며, React 생태계를 활용하고자 한다.

Next.js 14는 두 가지 라우팅 시스템을 제공한다:
- **Pages Router**: 기존 방식. `pages/` 디렉토리 기반, `getServerSideProps`/`getStaticProps` 패턴
- **App Router**: Next.js 13+에서 도입. `app/` 디렉토리 기반, React Server Components, 중첩 레이아웃, 서버 액션 지원

## Decision

**App Router를 사용한다.**

- React Server Components로 서버에서 직접 데이터를 fetching하여 클라이언트 번들 크기를 줄일 수 있다
- 중첩 레이아웃(`layout.tsx`)으로 `(browse)/[locale]/` 구조의 locale 기반 라우팅을 자연스럽게 구현할 수 있다
- 라우트 그룹 `(browse)`으로 URL에 영향 없이 레이아웃을 공유할 수 있다
- Next.js의 공식 권장 방향이 App Router이며, 새로운 기능(Server Actions, Parallel Routes 등)이 App Router에서만 지원된다

## Consequences

**긍정적:**
- 서버 컴포넌트로 인한 성능 이점 (클라이언트 JS 감소)
- 중첩 레이아웃으로 IntlProvider → UserProvider → Header/Footer 구조를 깔끔하게 구성
- `revalidatePath()`를 통한 세밀한 캐시 무효화

**부정적:**
- Pages Router 대비 학습 곡선이 높음
- 서버/클라이언트 컴포넌트 경계를 명확히 관리해야 함 (`'use client'` 지시어)
- 일부 서드파티 라이브러리의 App Router 호환성 이슈 가능
