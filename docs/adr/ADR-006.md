# ADR-006: Zod 스키마 검증 전략

## Status

승인 (Accepted) — 2023-11-24

## Context

API 라우트에서 사용자 입력(회원가입, 글 작성, 댓글 등)을 검증해야 한다. 검증 실패 시 구체적인 에러 메시지를 반환해야 하며, TypeScript 타입과 검증 로직의 일관성이 중요하다.

고려한 대안:
- **Joi**: 성숙한 검증 라이브러리지만 TypeScript 타입 추론 미지원
- **Yup**: Formik과 잘 맞지만 TypeScript 통합이 Zod보다 약함
- **class-validator**: 데코레이터 기반. NestJS 스타일이며 Next.js API Routes와 맞지 않음
- **직접 구현**: 간단하지만 에러 메시지 표준화가 어려움

## Decision

**Zod를 사용하여 API 입력을 검증하고, `.safeParse()`로 에러를 처리한다.**

- `src/validation/schema.ts`에 모든 검증 스키마를 중앙 관리
- `.safeParse()` 패턴으로 예외 없이 결과 확인
- `ApiResponse.badRequest()`가 `ZodError`의 issue 메시지를 자동 추출하여 응답
- 스키마: `userRegisterSchema`, `articleInputSchema`, `articleUpdateSchema`, `commentSchema`

## Consequences

**긍정적:**
- 스키마에서 TypeScript 타입 자동 추론 (`z.infer<typeof schema>`)
- `.safeParse()`로 try-catch 없이 깔끔한 에러 핸들링
- ZodError가 구조화된 에러 정보를 제공하여 ApiResponse와 통합 용이
- 스키마 합성(`merge`, `extend`)으로 재사용 가능 (articleUpdateSchema = articleInputSchema + slug)

**부정적:**
- 클라이언트 사이드 검증에는 별도 로직 필요 (현재 서버 검증만 사용)
- 복잡한 조건부 검증(필드 간 의존)은 `.refine()`으로 처리해야 하며 가독성 저하
