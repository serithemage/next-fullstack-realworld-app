# 바이브 코딩 마이그레이션 매뉴얼

> 기존 프로젝트를 바이브 코딩(AI 지원 개발)에 적합한 프로젝트로 전환하는 단계별 가이드.
> 실제 프로젝트 전환 과정에서 사용된 프롬프트와 결과를 기반으로 작성되었다.

---

## 개요

**바이브 코딩** 이란 AI 코딩 에이전트(Claude Code 등)를 활용하여 개발 생산성을 극대화하는 개발 방식이다. AI가 코드를 효과적으로 이해하고 수정하려면 프로젝트에 충분한 맥락(문서, 테스트, 컨벤션)이 갖춰져 있어야 한다.

이 매뉴얼은 **테스트도 문서도 없는 기존 프로젝트** 를 바이브 코딩에 적합한 상태로 전환하는 전 과정을 다룬다.

### 전환 전후 비교

| 항목 | 전환 전 | 전환 후 |
|------|---------|---------|
| CLAUDE.md | 없음 | 아키텍처, 빌드 명령, 코드 스타일 문서화 |
| ADR | 없음 | 10건 작성 |
| 테스트 | 0% | 테스트 인프라 + 전략 수립 |
| GitHub 이슈 | 없음 | Epic + 작업 이슈 체계 |
| Claude Code 스킬 | 없음 | 7종+ 프로젝트 전용 스킬 |
| 코드 품질 | 미검증 | 코드 리뷰 → 보안 이슈 3건 수정 |
| 자동화 | 없음 | Git hooks + 품질 게이트 |

### 전환 순서

```
Phase 1: 프로젝트 분석 및 문서화
    ↓
Phase 2: GitHub 이슈 체계화
    ↓
Phase 3: Claude Code 스킬 생성
    ↓
Phase 4: 테스트 인프라 구축
    ↓
Phase 5: 코드 리뷰 및 보안 이슈 수정
    ↓
Phase 6: 품질 자동화 (Git Hooks)
```

## Phase 1: 프로젝트 분석 및 문서화

> AI가 프로젝트를 이해할 수 있도록 맥락을 문서화하는 단계.
> CLAUDE.md는 AI 에이전트가 프로젝트에서 작업할 때 참조하는 핵심 문서다.

### Step 1-1: CLAUDE.md 자동 생성

Claude Code에는 CLAUDE.md를 자동 생성하는 내장 프롬프트가 있다. `/init` 명령을 사용하면 된다.

> **프롬프트:**
>
> ```
> /init
> ```

**결과:**

| 생성 파일 | 내용 |
|-----------|------|
| `CLAUDE.md` | 빌드 명령, 아키텍처 개요, 라우팅 구조, API 패턴, 인증, DB 스키마, 코드 스타일 |

**범용 팁:** `/init`이 생성하는 CLAUDE.md는 출발점이다. 프로젝트 고유의 컨벤션, 주의사항, 자주 쓰는 명령어를 추가로 보강하면 AI의 코드 품질이 크게 향상된다.

### Step 1-2: 심층 인터뷰를 통한 전환 계획 수립

AI에게 프로젝트 전환 계획을 수립하도록 요청한다. **심층 인터뷰** 를 요청하면 AI가 필요한 맥락을 질문하여 수집한다.

> **프롬프트:**
>
> ```
> 이 프로젝트를 바이브 코딩을 위한 프로젝트로 전환하려고 해.
> 문서화 및 테스트, CI/CD까지 진행할 거야.
> 이를 위한 심층 인터뷰를 진행해줘.
> ```

**결과:**

AI가 다음 항목에 대해 질문을 통해 맥락을 수집한다:

- 프로젝트 용도 (학습용/프로덕션)
- 배포 환경 (Vercel/AWS 등)
- 문서 언어 (한국어/영어)
- 테스트 전략 선호도
- CI/CD 요구사항

| 생성 파일 | 내용 |
|-----------|------|
| `docs/vibe-coding-transition-plan.md` | 전환 계획 전체 (1단계: 문서화 → 2단계: 테스트 → 3단계: CI/CD) |

**범용 팁:** "심층 인터뷰를 진행해줘"라는 프롬프트는 AI가 스스로 질문을 던져 맥락을 수집하게 하는 강력한 패턴이다. 요구사항이 불명확할 때 특히 유용하다.

### Step 1-3: OpenSpec을 사용한 체계적 문서 생성

OpenSpec은 변경을 proposal → design → tasks로 관리하는 spec-driven 워크플로우다.

> **프롬프트:**
>
> ```
> /opsx:propose 이 프로젝트 문서화
> ```

> **구현 실행:**
>
> ```
> /opsx:apply
> ```

> **완료 후 보관:**
>
> ```
> /opsx:archive
> ```

**결과:**

| 생성 파일 | 내용 |
|-----------|------|
| `CLAUDE.md` | 아키텍처, 빌드 명령, 코드 스타일 종합 문서 |
| `docs/code-conventions.md` | 코드 컨벤션 상세 |
| `docs/adr/ADR-001~010.md` | 아키텍처 결정 기록 10건 |
| `docs/adr/README.md` | ADR 목차 |
| `docs/vibe-coding-workflow.md` | 바이브 코딩 워크플로우 가이드 |

**커밋:** `03d89ed docs: 바이브 코딩 프로젝트 전환 1단계 - 종합 문서화`

### Step 1-4: 문서 리뷰

생성된 문서의 품질을 검증한다.

> **프롬프트:**
>
> ```
> 현재 만들어진 문서나 문서 구조에 대해서 리뷰해줘.
> ```

**범용 팁:** AI가 생성한 문서는 반드시 리뷰 단계를 거쳐야 한다. 구조적 문제, 누락된 내용, 불일치를 잡아낼 수 있다.

## Phase 2: GitHub 이슈 체계화

> 전환 계획을 실행 가능한 GitHub 이슈로 분해하는 단계.

### Step 2-1: 전환 계획을 이슈로 등록

> **프롬프트:**
>
> ```
> docs/vibe-coding-transition-plan.md를 GitHub 이슈로 등록해줘.
> 이슈에는 작업 배경, 개요, 인수조건이 들어가야 하고,
> 다른 작업과 의존 관계가 있을 경우 의존 관계도 기술해줘.
> ```

**결과:**

| 이슈 | 내용 | 라벨 |
|------|------|------|
| #1 | Epic: 바이브 코딩 프로젝트 전환 | `epic` |
| #2~#8 | 개별 작업 이슈 (테스트, CI/CD 등) | `testing`, `infra` 등 |

**범용 팁:** 이슈에 **인수조건** 을 명시하면 나중에 AI가 작업 완료 여부를 자동 검증할 수 있다. 이 프로젝트에서는 이를 스킬로 자동화했다(Phase 5 참조).

## Phase 3: Claude Code 스킬 생성

> 반복 작업을 자동화하는 프로젝트 전용 Claude Code 스킬을 만드는 단계.

### Step 3-1: 프로젝트 문서 기반 스킬 생성

> **프롬프트:**
>
> ```
> 만들어진 문서들을 이용해서 이 프로젝트에서 필요한 클로드 스킬을
> 공식문서에 따라 만들어줘. 궁금한 게 있으면 심층 인터뷰를 진행해.
> ```

초기 생성 결과가 너무 큰 스킬 파일을 만들었을 때:

> **보완 프롬프트:**
>
> ```
> 핵심 지시사항만 SKILL.md에 넣고 상세 내용은 기존에 작성된 문서들을
> 링크해서 사용해.
> ```

**결과:**

| 스킬 | 용도 |
|------|------|
| `openspec-propose` | OpenSpec 변경 제안 |
| `openspec-apply` | OpenSpec 태스크 구현 |
| `openspec-explore` | 아이디어 탐색 |
| `openspec-archive` | 완료된 변경 보관 |
| `gen-test` | ADR-007 패턴에 따른 테스트 생성 |
| `i18n-sync` | 다국어 번역 키 동기화 |
| `manage-issues` | GitHub 이슈 관리 |

**커밋:** `230cf04 docs: Claude Code 프로젝트 스킬 7종 추가 및 가이드 문서화`

### 스킬 작성 시 주의사항

- SKILL.md는 **500줄 미만** 으로 유지
- 상세 내용은 기존 문서를 참조하도록 지시
- `user-invokable: true` (오타 주의: `invocable`이 아님)
- `allowed-tools`는 미지원 속성이므로 사용 금지

## Phase 4: 테스트 인프라 구축

> 테스트가 전무한 상태에서 테스트 체계를 수립하는 단계.

### Step 4-1: 테스트 전략 수립 (심층 인터뷰)

> **프롬프트:**
>
> ```
> 이 프로젝트에 테스트를 추가하고 싶어.
> Unit, Integration, E2E, Lint까지 테스트를 구현해줘.
> 추가적으로 맥락을 요청하려면 심층 인터뷰를 진행해.
> ```

**결과:**

AI가 다음을 질문하여 테스트 전략을 수립:

- 우선순위 모듈 (API routes vs 컴포넌트)
- 커버리지 목표 (80% 등)
- E2E 도구 선호 (Playwright vs Cypress)
- CI/CD 연동 방식

| 생성 파일 | 내용 |
|-----------|------|
| `docs/adr/ADR-007.md` | 테스트 전략 결정 기록 |
| `docs/adr/ADR-008.md` | E2E 테스트 전략 |

### Step 4-2: 테스트 구현 계획 실행

계획이 수립되면 plan mode로 실행한다.

> **프롬프트 (plan mode 진입 후):**
>
> ```
> 테스트 인프라 + 테스트 구현 계획을 실행해줘.
> ```

**범용 팁:** 대규모 작업은 plan mode(`/plan`)를 사용하여 계획을 먼저 수립하고 승인 후 실행하는 것이 안전하다.

## Phase 5: 코드 리뷰 및 보안 이슈 수정

> AI를 활용한 코드 리뷰로 보안 취약점을 발견하고 수정하는 단계.

### Step 5-1: 코드 리뷰 요청

> **프롬프트:**
>
> ```
> 이 프로젝트를 리뷰하고 개선할 점을 제안해줘.
> ```

**결과:** Critical 3건, High 4건, Medium 5건의 이슈 발견.

### Step 5-2: 발견된 이슈를 GitHub에 등록 + 이슈 관리 스킬 생성

> **프롬프트:**
>
> ```
> 발견된 이슈들을 GitHub 이슈에 등록해줘.
> 앞서서 이슈 등록에 사용한 프롬프트로 GitHub 이슈 관리 스킬을 만들어줘.
> ```

**결과:**

| 이슈 | Severity | 내용 |
|------|----------|------|
| #9 | Critical | XSS 취약점 — ArticleBody에 sanitizer 미적용 |
| #10 | Critical | IDOR 취약점 — PUT /api/user에 인증 없음 |
| #11 | Critical | throw Error — unfavorite 시 500 반환 |

| 생성 스킬 | 용도 |
|-----------|------|
| `manage-issues` | 이슈 생성/업데이트/닫기 템플릿 + 인수조건 검증 |

### Step 5-3: Critical 이슈 수정

> **프롬프트:**
>
> ```
> 크리티컬 이슈들에 대해서 작업 계획을 세우고 병렬로 작업을 진행해.
> ```

**결과:**

| 이슈 | 수정 내용 | 변경 파일 |
|------|----------|-----------|
| #9 XSS | `isomorphic-dompurify`로 HTML sanitize | `ArticleBody.tsx`, `package.json` |
| #10 IDOR | `getCurrentUser()` 인증 + 세션 ID + Zod 검증 | `user/route.ts`, `schema.ts` |
| #11 Error | `throw Error` → `ApiResponse.unauthorized()` | `favorite/route.ts` |

**커밋:** `ed7f858 fix: critical 보안 이슈 3건 수정 (#9, #10, #11)`

### Step 5-4: 인수조건 검증 후 이슈 닫기

> **프롬프트:**
>
> ```
> 인수조건도 만족시켰는지 확인해.
> ```

> **프롬프트:**
>
> ```
> 작업한 이슈들 인수조건 확인하고 충족 시 닫아줘.
> ```

**결과:** 12개 인수조건 중 12개 충족 확인 → 이슈 #9, #10, #11 모두 Close.

### Step 5-5: 스킬 개선 (학습된 패턴 반영)

작업 과정에서 "인수조건 검증을 자동화해야 한다"는 교훈을 얻어 스킬을 업데이트했다.

> **프롬프트:**
>
> ```
> 작업 완료 후 이슈 업데이트 시에는 인수조건을 확인하도록 스킬을 업데이트해.
> ```

**결과:** `manage-issues` 스킬에 인수조건 검증 단계 추가.

**커밋:** `2c0a1a2 docs: manage-issues 스킬에 인수조건 검증 단계 추가`

## Phase 6: 품질 자동화 (Git Hooks)

> 커밋/푸시 시 자동으로 품질 검증이 실행되도록 설정하는 단계.

### Step 6-1: Git Hook 설정

> **프롬프트:**
>
> ```
> Git Hook을 사용해서 테스트나 검증이 상시 동작하도록 설정하고 싶어.
> 어떻게 하는 게 좋을지 안을 제시해줘.
> 맥락을 보강하기 위해서 심층 인터뷰를 사용해.
> ```

**결과:**

| Hook | 실행 내용 |
|------|----------|
| `pre-commit` | lint + 타입 체크 |
| `pre-push` | 테스트 실행 |

### Step 6-2: 문서 변경 시 바이패스

> **프롬프트:**
>
> ```
> 문서와 같은 테스트와 직접 상관 없는 파일이 변경될 경우
> 검증을 바이패스하도록 해줘.
> ```

### Step 6-3: Hook 동작 검증

> **프롬프트:**
>
> ```
> 훅에 의한 테스트가 잘 동작하는지 확인해줘.
> 문서 변경이 바이패스되는 것도 확인되어야 해.
> ```

## 핵심 교훈

### 효과적인 프롬프트 패턴

| 패턴 | 설명 | 예시 |
|------|------|------|
| **심층 인터뷰** | AI가 질문을 통해 맥락을 수집 | "심층 인터뷰를 진행해줘" |
| **병렬 작업 지시** | 독립 작업을 동시 실행 | "병렬로 작업을 진행해" |
| **인수조건 명시** | 완료 기준을 검증 가능하게 | 이슈에 체크리스트 포함 |
| **스킬 자동 생성** | 반복 패턴을 스킬로 추출 | "사용한 프롬프트로 스킬을 만들어줘" |
| **리뷰 요청** | 생성물의 품질 검증 | "리뷰하고 개선할 점을 제안해줘" |
| **Plan mode** | 대규모 작업 전 계획 수립 | 자동 진입 또는 `/plan` |

### 주의사항

1. **문서화가 먼저다** — AI가 코드를 이해하려면 CLAUDE.md, ADR, 코드 컨벤션이 필요하다. 테스트나 기능 구현보다 문서화를 먼저 해야 한다.

2. **SKILL.md는 작게 유지** — 500줄 미만으로 핵심 지시만 넣고, 상세 내용은 기존 문서를 참조하게 한다.

3. **인수조건을 반드시 넣어라** — 이슈에 인수조건이 있으면 AI가 작업 완료를 자동 검증할 수 있다. 없으면 "완료"의 기준이 모호해진다.

4. **심층 인터뷰를 적극 활용하라** — 요구사항이 불명확할 때 "심층 인터뷰를 진행해줘"라고 하면 AI가 체계적으로 질문하여 요구사항을 구체화한다.

5. **생성 후 반드시 리뷰** — AI가 생성한 문서, 코드, 스킬은 반드시 리뷰 단계를 거쳐야 한다.

6. **학습된 패턴을 스킬로 추출** — 작업 중 발견한 유용한 패턴은 즉시 Claude Code 스킬로 만들어 재사용한다.
